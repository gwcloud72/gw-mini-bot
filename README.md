 미니챗

> 프런트엔드와 백엔드 서버를 분리해 구성한 SSE 기반 웹 채팅 프로젝트

**서비스:** https://gwcloud72.github.io/gw-mini-bot/

## 기획 의도

대화형 서비스에서는 답변 내용뿐 아니라 **답변을 기다리는 과정도 자연스럽게 보여야 한다**고 생각했습니다.

외부 대화 API의 응답이 늦어질 때 화면이 멈춘 것처럼 보이거나, 답변이 여러 문장씩 한꺼번에 나타나는 문제를 직접 해결해 보기 위해 미니챗을 기획했습니다.

단순히 질문과 답변을 표시하는 채팅 화면을 만드는 데 그치지 않고, 프런트엔드와 백엔드 서버의 역할을 분리해 민감한 인증 정보를 브라우저에 노출하지 않으면서 실시간 응답을 전달하는 것을 목표로 했습니다.

또한 대화 기록, 재시도, 응답 상태, 모바일 화면, 브라우저별 애니메이션 차이까지 함께 다뤄 작은 규모에서도 실제 서비스처럼 사용할 수 있는 채팅 경험을 구현하고자 했습니다.

## 프로젝트 소개

미니챗은 사용자가 입력한 질문을 백엔드 서버로 전달하고, 외부 대화 API에서 생성되는 답변을 SSE 방식으로 실시간 중계하는 웹 애플리케이션입니다.

프런트엔드는 화면 구성과 대화 상태 관리를 담당하고, 백엔드 서버는 요청 검증, 외부 API 연결, 스트리밍 응답 중계, 일일 사용량 제한을 담당합니다.

```text
사용자 브라우저
→ React 프런트엔드
→ 백엔드 서버
→ 외부 대화 API
→ SSE 스트리밍 응답
```

## 주요 기능

- SSE 기반 실시간 답변
- 답변 준비·생성·완료 상태 표시
- 대화 기록 브라우저 저장
- 새 대화 및 답변 재시도
- 답변 복사
- Markdown과 코드 블록 표시
- 계절별 화면 스킨
- 데스크톱·모바일 반응형 화면
- 동작 줄이기 설정과 브라우저 호환 처리
- 요청 데이터 검증
- IP 기준 일일 사용량 제한

## 기술 구성

| 구분 | 사용 기술 |
|---|---|
| 프런트엔드 | React, TypeScript, Tailwind CSS, Vite |
| 실시간 통신 | Server-Sent Events |
| 백엔드 | TypeScript 기반 API 서버 |
| 상태 저장 | localStorage |
| 프런트 배포 | GitHub Pages |
| 품질 검증 | ESLint, TypeScript, Vitest, GitHub Actions |

## 구조

기능이 한 파일에 집중되지 않도록 화면, 상태 관리, 서버 통신의 책임을 분리했습니다.

```text
App
└─ ChatRoom
   ├─ ChatHeader
   ├─ MessageList
   ├─ MessageBubble
   ├─ ChatComposer
   └─ ChatMenu

hooks
└─ 대화 상태와 스트림 생명주기 관리

services
└─ 백엔드 요청과 SSE 응답 처리

shared
└─ 프런트엔드와 백엔드가 함께 사용하는 통신 규격
```

CSS는 임시 덮어쓰기를 반복하기보다 역할이 분명한 클래스와 의미 기반 변수를 사용했습니다. `!important`와 중복 선택자를 추가하지 않고, 기존 환경변수와 스트리밍 이벤트 규격을 임의로 변경하지 않는 것을 기준으로 삼았습니다.

# 트러블슈팅

## 1. 답변을 기다리는 동안 화면이 멈춘 것처럼 보인 문제

### 문제 상황

사용자가 메시지를 전송한 뒤 첫 답변이 도착하기 전까지 화면에 변화가 없어, 실제 요청이 처리 중이어도 서비스가 멈춘 것처럼 보였습니다.

### 원인 분석

백엔드 서버가 외부 대화 API의 응답을 먼저 기다린 뒤 브라우저에 SSE 응답을 반환하고 있었습니다.

```text
사용자 요청
→ 외부 대화 API 연결 대기
→ 외부 대화 API 응답
→ 브라우저 SSE 연결 시작
```

이 구조에서는 외부 API의 첫 응답 시간이 그대로 빈 대기 시간으로 노출됐습니다.

프런트엔드는 답변 생성 상태를 표시할 준비가 되어 있었지만, 백엔드 서버에서 상태 이벤트를 전달하기 전까지 화면을 변경할 근거가 없었습니다.

### 해결 과정

프런트엔드와 백엔드 서버가 공유하는 스트리밍 이벤트 순서를 다음과 같이 정리했습니다.

```text
ready
→ generating
→ chunk
→ done 또는 error
```

백엔드 서버는 외부 API의 전체 응답을 기다리지 않고 브라우저와의 SSE 연결을 먼저 열도록 변경했습니다.

프런트엔드는 수신한 이벤트에 따라 다음 상태를 단계적으로 표시했습니다.

```text
서버에 연결 중…
답변 준비 중…
답변 작성 중…
```

또한 완료되지 않은 이전 요청이 뒤늦게 도착해 새로운 대화 상태를 덮어쓰지 않도록, 현재 활성화된 스트림을 구분해 처리했습니다.

### 결과

답변이 생성되기 전에도 사용자가 현재 처리 상태를 확인할 수 있게 됐습니다.

스트림이 중간에 종료되거나 오류가 발생한 경우에는 무한 대기하지 않고 오류 상태와 재시도 기능을 제공할 수 있게 됐습니다.

### 배운 점

SSE를 사용한다는 사실만으로 자연스러운 실시간 경험이 만들어지는 것은 아니었습니다.

연결을 시작하는 시점과 이벤트 순서를 프런트엔드와 백엔드 서버가 동일하게 이해해야, 사용자에게 정확한 진행 상태를 전달할 수 있다는 점을 배웠습니다.

## 2. 답변이 끊기거나 여러 문장씩 한꺼번에 나타난 문제

### 문제 상황

답변 데이터는 계속 도착하고 있었지만 문장이 일정하지 않은 간격으로 나타났고, 긴 답변에서는 여러 글자가 한 번에 표시되는 현상이 발생했습니다.

### 원인 분석

응답 조각이 도착할 때마다 다음 작업이 반복되고 있었습니다.

```text
메시지 상태 갱신
→ 대화 목록 재렌더링
→ Markdown 처리
→ 스크롤 높이 계산
→ 자동 스크롤
```

짧은 시간 안에 상태 변경과 화면 계산이 반복되면서 브라우저가 일부 렌더링 작업을 한꺼번에 처리했습니다.

서버에서는 정상적으로 스트리밍하고 있었지만, 프런트엔드의 화면 갱신 방식 때문에 사용자는 답변이 끊기는 것으로 느낄 수 있었습니다.

### 해결 과정

- 현재 스트리밍 중인 메시지만 갱신
- 완료된 이전 메시지의 불필요한 재렌더링 차단
- 스트리밍 중에는 일반 텍스트로 표시
- 응답 완료 후 Markdown 렌더링
- 같은 화면 프레임에 도착한 응답 조각을 묶어서 반영
- 사용자가 대화 하단을 보고 있을 때만 자동 스크롤
- 실제 서버 응답을 인위적으로 늦추는 타자 효과 제거

### 결과

답변이 덩어리 형태로 갑자기 나타나는 현상이 줄었고, 긴 답변에서도 스크롤과 입력창의 반응을 비교적 안정적으로 유지할 수 있게 됐습니다.

### 배운 점

체감 속도 문제를 서버 응답 속도 하나로만 판단하면 정확한 원인을 찾기 어려웠습니다.

네트워크 데이터 수신, React 상태 갱신, Markdown 렌더링, 자동 스크롤을 각각 분리해서 확인해야 실제 병목을 찾을 수 있다는 점을 배웠습니다.

## 3. 로컬에서는 정상인데 배포 환경에서 설정 오류가 발생한 문제

### 문제 상황

로컬 타입 검사와 빌드는 정상적으로 완료됐지만, 실제 배포 환경에서는 필수 설정 누락이나 배포 대상 불일치 오류가 발생했습니다.

프런트엔드에서 사용하는 공개 설정과 백엔드 서버에서 사용하는 민감한 설정이 모두 환경변수라는 이름으로 관리돼, 각 값의 역할과 등록 위치가 혼동되기 쉬웠습니다.

### 원인 분석

설정값은 사용 목적에 따라 적용되는 위치가 달랐습니다.

```text
프런트엔드 공개 설정
→ 백엔드 주소, 요청 제한 시간, 메시지 길이

백엔드 비공개 설정
→ 외부 API 키, 인증 정보, 식별용 비밀값

서버 실행 설정
→ 허용 Origin, 서비스 이름, 데이터 저장 연결
```

빌드 시점에 필요한 값과 실제 서버 실행 시점에 필요한 값을 구분하지 않으면, 로컬에서는 통과해도 실제 서비스에서는 오류가 발생할 수 있었습니다.

### 해결 과정

설정을 역할에 따라 분리했습니다.

- 브라우저에 노출돼도 되는 값만 프런트엔드 공개 설정으로 관리
- API 키와 인증 정보는 백엔드 서버의 비공개 설정으로 관리
- 허용 Origin과 서버 연결 정보는 서버 설정 파일에서 관리
- 기존 환경변수 이름과 운영값은 임의로 변경하지 않음
- 배포 전에 필수 설정, 서버 주소, 허용 Origin, 통신 규격을 검사

### 결과

공개 저장소에 민감한 값이 포함되지 않으면서도, 설정 누락과 잘못된 배포 대상을 실제 서비스 반영 전에 확인할 수 있게 됐습니다.

프런트엔드와 백엔드 서버에서 각각 어떤 설정을 책임지는지도 명확해졌습니다.

### 배운 점

환경변수는 단순히 값을 외부로 분리하는 기능이 아니라, 프런트엔드와 백엔드 서버 사이의 실행 환경을 정의하는 계약이라는 점을 배웠습니다.

기존 변수 이름을 안정적으로 유지하면서 공개 설정과 비공개 설정의 책임을 분리하는 것이 중요했습니다.

## 배운 점

이번 프로젝트를 통해 다음 기준을 정리했습니다.

1. 프런트엔드와 백엔드 서버의 통신 규격을 먼저 고정한다.
2. 외부 API 지연과 프런트엔드 렌더링 지연을 분리해서 분석한다.
3. 스트리밍 중에는 현재 답변에 필요한 영역만 갱신한다.
4. 환경변수 이름과 API 규격을 임의로 변경하지 않는다.
5. 공개 설정과 비공개 설정의 책임을 명확하게 구분한다.
6. 로컬 검사뿐 아니라 실제 배포 로그와 브라우저 동작까지 확인한다.
7. 애니메이션은 화려함보다 응답 상태 전달과 브라우저 호환성을 우선한다.

미니챗은 작은 규모의 프로젝트지만, 실시간 통신을 구현하면서 프런트엔드 상태 관리, 백엔드 서버의 스트리밍 중계, 배포 설정과 보안의 경계를 함께 경험할 수 있었던 프로젝트입니다.
